## Этап 1: Подготовка к работе

Для начала необходимо подготовить рабочее окружение: завести аккаунт на платформе GitHub и установить консольную утилиту Git.

### 1.1. Регистрация на платформе GitHub

**GitHub** — это крупнейший облачный сервис для хостинга кода и совместной работы над IT-проектами.

1. Откройте сайт [github.com](https://github.com/).
2. В правом верхнем углу нажмите **"Sign up"**.
3. Пройдите шаги регистрации: укажите актуальную электронную почту, придумайте надежный пароль и уникальный никнейм.
4. Обязательно подтвердите создание профиля, перейдя по ссылке из письма, которое придет на ваш email.

### 1.2. Установка и базовая настройка Git

**Git** — локальная система контроля версий, которая будет отслеживать все изменения в файлах вашего проекта.

1. Перейдите на сайт разработчиков: [git-scm.com](https://git-scm.com/).
2. Загрузите инсталлятор, подходящий для вашей операционной системы (Windows, macOS или Linux).
3. Запустите установку. Если вы новичок, просто нажимайте "Next", оставляя настройки по умолчанию.
4. Откройте консоль (например, Git Bash в Windows) и проверьте успешность установки:

**Проверка версии:**
```bash
$ git version
git version 2.51.0.windows.1
```
**Важно:** Теперь нужно представиться системе, чтобы авторство ваших изменений фиксировалось корректно.
```bash
$ git config --global user.name "testik"
$ git config --global user.email "testik@gmail.com"
```
## Этап 2: Управление проектом на вашем компьютере

В этом разделе разобраны основные команды для ведения локального репозитория.

```bash
$ cd "C:/Users/Topo/Laba2"
```

### 2.1. Создание репозитория и просмотр статуса

**`git init`**: Превращает обычную папку в репозиторий Git.
```bash
$ git init
Initialized empty Git repository in C:/Users/Topo/Laba2/.git/
```
Введите команду **`ls -la`**, чтобы убедиться, что репозиторий создан (наличие .git).

``` bash
$ ls -la
total 16
drwxr-xr-x 1 testik 197121   0 Sep 15 14:10 ./
drwxr-xr-x 1 testik 197121   0 Sep 15 14:05 ../
drwxr-xr-x 1 testik 197121   0 Sep 15 14:10 .git/
-rw-r--r-- 1 testik 197121 142 Sep 15 14:08 ReadMe.md
```

**`git status`**: Выводит текущее состояние файлов в папке (есть ли изменения, которые еще не сохранены).
```bash
$ git status
On branch main
No commits yet
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        ReadMe.md
nothing added to commit but untracked files present (use "git add" to track)
```
### 2.2. Работа с индексом, коммитами и историей

**`git add .`**: Добавляет сразу все измененные и новые файлы в индекс (готовит к коммиту)

```bash
$ git add .
```

**`git add <имя_файла>`**: Если нужно добавить только конкретный файл

```bash
$ git add ReadMe.md
```
**`git reset <имя_файла>`**: Убирает файл из индекса, если вы передумали его коммитить (сам файл при этом не удаляется).

```bash
$ git reset config.json
```
**`git commit -m "Сообщение"`**: Делает коммит кода. Сообщение должно отражать изменения.

```bash
$ git commit -m "Коммит 1 (master) написание  1 главы"
[master 1002591] Коммит 1 (master) написание  1 главы
 1 file changed, 29 insertions(+)
```
**`git commit --amend -m "Сообщение"`**: Позволяет переписать комментарий к самому последнему коммиту, если вы допустили ошибку.
```bash
$ git commit --amend -m "Коммит 2 (master) написание  1 главы"
```

**`git log`** — Показывает журнал коммитов.

### 2.3. Ветвление проекта

**`git branch`**: Выводит список существующих веток. Звездочка означает текущую.

```bash
$ git branch
  master
* local
  global
  ```

**`git branch <имя_ветки>`**: Создает новую ветку для изолированной разработки.

```bash
$ git branch global
```

**`git checkout <имя_ветки>`** (или **`git switch <имя_ветки>`**): Выполняет переход в указанную ветку.

```bash
$ git checkout global
Switched to branch 'global'
```

**`git merge <имя_ветки>`**: Интегрирует изменения из указанной ветки в ту, в которой вы сейчас находитесь.

```bash
$ git merge global
```
При слиянии иногда возникают конфликты, если одни и те же строки были изменены по-разному:
```bash
$ git merge global
Auto-merging ReadMe.md
CONFLICT (content): Merge conflict in ReadMe.md
Automatic merge failed; fix conflicts and then commit the result.
```
В таком случае нужно открыть файл, руками оставить нужный вариант кода, после чего сделать **`git add`** и **`git commit`**.

**`git checkout <хэш_коммита>`**: Позволяет "отмотать время назад" и посмотреть проект в том виде, в котором он был в момент конкретного коммита.

```bash
$ git checkout a1b2c3d
Note: switching to 'a1b2c3d'.
You are in 'detached HEAD' state.
```

**`git checkout -b <имя_ветки>`** (или **`git switch -c <имя_ветки>`**):  Быстрая команда: создает ветку и моментально переключает вас на нее.

```bash
$ git checkout -b bugfix
Switched to a new branch 'bugfix'
```

## Этап 3: Совместная работа и удалённые репозитории 

Чтобы не потерять код и работать в команде, локальный репозиторий связывают с облачным сервером.

### 3.1. Связь локального кода с сервером

Сначала создайте пустой репозиторий в вашем профиле на GitHub (без файлов README или .gitignore).

**`git remote add <имя> <URL>`**: Подключает удаленный сервер к вашему проекту на компьютере.
* `<имя>` — алиас сервера, общепринятый стандарт — origin.
* `<URL>` — ссылка на ваш репозиторий.

```bash
$ git remote add origin https://github.com/GmDevTeam/topo2.git
```

**`git push -u <имя_сервера> <имя_ветки>`**: Выгружает ваши локальные коммиты на сервер. Флаг -u запоминает связь, чтобы потом писать просто git push.

```bash
$git push -u origin master
```
**`git push --all origin`**: Отправляет абсолютно все локальные ветки и их историю в удаленный репозиторий.

```bash
git push --all origin
```
**`git pull`**: Скачивает новые изменения с сервера и автоматически сливает их с вашей активной веткой.

```bash
git pull origin master
```

**`git clone <URL>`**: Загружает чужой или свой проект с сервера на компьютер "с нуля", сохраняя всю историю версий.

### 3.2. Работа с Fork и Pull Request

Fork (Форк) — это кнопка на GitHub, которая делает абсолютную копию чужого публичного репозитория в ваш личный профиль. Это позволяет вам вносить любые правки в код, не опасаясь сломать проект автора.
Pull Request (PR) — это механизм предложений. Когда вы улучшили код в своем форке, вы создаете PR, чтобы разработчик оригинального проекта проверил ваши изменения и, если все хорошо, добавил их в основную кодовую базу.
```bash
$ git clone https://github.com/Ban049/CloneFor2topo
Cloning into '123.txt'...
remote: Enumerating objects: 19, done.
remote: Counting objects: 100% (3/3), done.
remote: Compressing objects: 100% (3/3), done.
remote: Total 19 (delta 0), reused 0 (delta 0), pack-reused 16 (from 1)
Receiving objects: 100% (19/19), 29.71 KiB | 239.00 KiB/s, done.
Resolving deltas: 100% (5/5), done.
```

**Алгоритм отправки Pull Request:**
1. Откройте страницу вашего форк-репозитория на GitHub.
2. Перейдите во вкладку "Pull requests" и кликните "New pull request".
3. Слева должен стоять оригинальный проект и ветка main, а справа — ваш форк и ветка с изменениями.
4. Заполните понятный заголовок и детальное описание того, какую проблему решает ваш код.
5. Нажмите зеленую кнопку "Create pull request".
